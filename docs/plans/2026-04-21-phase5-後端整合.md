# Phase 5 — 後端整合（前端側）

## Context

Phase 1–4 已完成前端全部功能，資料層以 `src/services/api.ts` 的 localStorage 模擬實作。
本階段將 `api.ts` 內部實作替換為呼叫 `speak-excel-api`（Express + Supabase），同時預留登入 / 部署配置的擴充點。

**前提與限制**

- 後端 contract 尚未定案 → 本 plan 以彈性設計為主，API 路由與錯誤格式採通用假設，後端定稿後可局部修正
- 本階段**不實作登入**，但在 `api.ts` / router / env 中預留介接點，未來可最小改動接入
- 依 `.claude/rules/api-design.md`：**`api.ts` 公開函式簽名不可變**（參數與回傳型別鎖定），頁面元件與 composables 不須修改
- 後端 base URL 由 `import.meta.env.VITE_API_BASE_URL` 讀取，開發預設 `http://localhost:3000/api`；部署環境由 `.env.production` 或 CI 注入

---

## 一、API Contract 假設（供前端先行）

採 REST 慣例，若後端實際路由不同，僅需調整 `api.ts` 內部 URL 常數。

| 前端函式                    | HTTP   | 路徑              | Request Body       | Response             |
| --------------------------- | ------ | ----------------- | ------------------ | -------------------- |
| `getChecklists()`           | GET    | `/checklists`     | —                  | `ChecklistSummary[]` |
| `getChecklist(id)`          | GET    | `/checklists/:id` | —                  | `Checklist` / 404    |
| `createChecklist(data)`     | POST   | `/checklists`     | `ChecklistInput`   | `Checklist`          |
| `updateChecklist(id, data)` | PATCH  | `/checklists/:id` | `ChecklistUpdate`  | `Checklist`          |
| `deleteChecklist(id)`       | DELETE | `/checklists/:id` | —                  | `204 No Content`     |
| `getGauges()`               | GET    | `/gauges`         | —                  | `Gauge[]`            |
| `createGauge(name)`         | POST   | `/gauges`         | `{ name: string }` | `Gauge` / 409 重複   |
| `deleteGauge(id)`           | DELETE | `/gauges/:id`     | —                  | `204 No Content`     |

**錯誤格式假設**：

```json
{ "error": "訊息內容", "code"?: "CHECKLIST_NOT_FOUND" }
```

前端統一在 `http.ts` 攔截非 2xx 狀態，丟出 `ApiError`（含 `status`、`code`、`message`）。

## 三、http.ts 設計

職責：

- 從 `import.meta.env.VITE_API_BASE_URL` 讀取 base URL（缺失時 throw 啟動期錯誤）
- 封裝 `request<T>(path, init)` → 回傳 Promise<T>
- 預設 `Content-Type: application/json`、處理 JSON 解析、處理 `204 No Content`
- **Auth 預留點**：
  - 保留 `getAuthToken()` hook（目前回傳 `undefined`），未來改讀 `localStorage['auth_token']` 或 Pinia store
  - 若 token 存在，自動注入 `Authorization: Bearer <token>`
  - 401 統一處理：觸發 `window.dispatchEvent(new CustomEvent('auth:unauthorized'))`，Phase 6 接登入時由 App.vue 監聽
- 錯誤正規化：非 2xx → 解析 JSON error body → throw `ApiError`
- Timeout：預設 10s（使用 `AbortController`）

---

## 四、api.ts 改寫重點

- 公開函式簽名維持不變（驗收條件）
- 移除 `readStore` / `writeStore` / `STORAGE_KEYS` / UUID 產生（改由後端給）
- 保留 `sanitizeChecklistName` / `sanitizeGaugeName` 作為前端 pre-flight 驗證（避免送出明顯無效請求），後端仍是最終權威
- 每個函式改為：

```ts
export async function getChecklists(): Promise<ChecklistSummary[]> {
  return request<ChecklistSummary[]>('/checklists')
}
```

- 排序邏輯（`updatedAt` desc、`name` zh-Hant）**改由後端負責**；前端保留 fallback 排序在 util，以防後端未處理

## 五、UI / UX 配套

由於網路呼叫會有延遲與失敗，頁面元件雖不改簽名，但需補強使用者反饋：

1. **Loading 狀態**：各 view 既有 `ref<boolean>` loading → 檢查目前是否都有覆蓋（列表、預覽、儲存按鈕 disabled）
2. **錯誤顯示**：統一以 `v-snackbar` 顯示 `ApiError.message`；建議於 `App.vue` 掛一個全域錯誤 bus（或 provide/inject）
3. **樂觀更新**：刪除量具、刪除檢查表不做樂觀更新（操作後重新 fetch 列表），避免與後端狀態分歧
4. **重試**：本階段不實作自動重試，錯誤訊息提示使用者手動重試即可

> 如發現哪個 view 尚未處理 error 狀態，plan 執行時建立獨立 sub-task。

## 六、資料遷移（localStorage → 後端）

- **不做自動遷移**：Phase 5 上線後舊 localStorage 資料視為作廢
- 可選擇性提供一次性 `scripts/migrate-localstorage.ts` 工具讓使用者匯出 JSON，再手動由後端 import；視後端是否需要決定是否做
- `.claude/rules/api-design.md` 要求「localStorage 階段 key 與後端 Schema 對齊」→ 若後端欄位命名不同，需在 `http.ts` 層做一次性 mapping

## 七、實作順序

### Step 1 — 基礎設施（先不動 api.ts）

1. 建立 `.env.development` / `.env.example`，文件化 `VITE_API_BASE_URL`
2. 建立 `src/services/apiError.ts`（`ApiError` class）
3. 建立 `src/services/http.ts`（request wrapper + auth 預留 + 401 event）
4. 撰寫 unit test：`http.ts` 的 error normalization、timeout、auth header 注入

### Step 2 — 切換 api.ts

5. 改寫 `api.ts` 全部函式為呼叫 `http.request`
6. 保留輸入驗證（`sanitizeXxxName`）

8. 手動驗證：`npm run build` 通過型別檢查

### Step 3 — UI 配套

9. 檢查各 View 的 loading / error 處理是否完備，補齊缺漏
10. 在 `App.vue` 建立全域 snackbar，訂閱錯誤事件

### Step 4 — 環境與部署準備

11. 更新 `docs/DEVELOPMENT.md` 補充 `VITE_API_BASE_URL` 與本地後端啟動說明
12. 更新 `docs/FEATURES.md` 將 Phase 5 標為 ✅
13. 更新 `docs/CHANGELOG.md`

### Step 5 — 登入預留驗證（不實作 UI）

14. 確認 `http.ts` 的 `getAuthToken()` hook、401 event、`Authorization` header 注入可用（單元測試）
15. 於 plan 尾端登記「Phase 6 登入功能」的 TODO，之後另建 plan

## 八、驗證方式

1. 本地啟動 `speak-excel-api` at `http://localhost:3000/api`
2. `npm run dev` 啟動前端
3. 走完完整流程：建立檢查表 → 編輯 → 儲存 → 預覽 → 匯出 → 刪除；量具新增 / 刪除
4. 模擬離線（關閉後端）→ 確認錯誤訊息顯示且不會白畫面
5. 修改 `VITE_API_BASE_URL` 指向部署環境 → 確認運作
6. `npm run build` 通過型別檢查
7. 既有單元測試不退化
